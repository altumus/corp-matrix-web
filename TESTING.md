# Ручное тестирование Corp Matrix

Этот документ — чек‑лист того, что **обязательно проверяется живым человеком на живом устройстве**. Всё, что закрывается Playwright‑скриптами (формы, CRUD, роутинг, настройки, keyboard shortcuts, DOM‑assertions, `axe-core`, network‑mocked edge‑cases), сюда не входит — оно живёт в `qa-agent/` и прогоняется в CI.

Принцип отбора: в список попадает только то, что проверяет либо **реальное оборудование** (камера, микрофон, динамики, гарнитура, тач, вибро), либо **системные интеграции** (iOS A2HS, Android SAF, push на устройстве, системные permission‑диалоги), либо **восприятие человеком** (читаемость, плавность, SAS‑эмодзи, screen reader).

## 0. Подготовка окружения

### 0.1. Обязательные реальные устройства

- **iPhone** (актуальная и предыдущая major iOS, минимум одно из них с notch)
- **Android‑телефон** (актуальный Chrome, средний ценовой сегмент)
- **Ноутбук/десктоп** с работающей web‑камерой и микрофоном
- **Bluetooth‑гарнитура / AirPods** (хотя бы одни)
- **Второй телефон или планшет** — для cross‑device и звонков

Эмуляторы (DevTools device mode, iOS Simulator) **не заменяют реальное устройство** для пунктов из этого документа. Их можно использовать для первичного debug, но финальный прогон — только на железе.

### 0.2. Homeserver для тестирования

Приложение — это чисто клиент; весь backend — Matrix homeserver (Synapse / Dendrite / любой другой). Для ручного тестирования нужен **один** сервер, к которому подключаются все устройства. Варианты:

**Вариант A (рекомендуется) — локальный Synapse в Docker.** Один сервер, все тестовые аккаунты в одной «организации», доступен всем устройствам по локальной сети.

**Вариант B — публичный тестовый homeserver (например, `matrix.org` или корпоративный).** Быстрее, но вы зависите от его доступности и не можете проверить админские вещи.

**Вариант C — два федеративных Synapse** (готовая конфигурация в `federation-test/docker-compose.yml`). Нужен только для сценариев cross‑org, в основной ручной прогон не входит.

#### 0.2.1. Вариант A: локальный Synapse

На машине, доступной с тестовых устройств по сети (ноутбук тестировщика, dev‑сервер и т. п.):

```bash
# Поднимаем одиночный Synapse (используем конфиг из federation-test,
# но запускаем только первый сервер)
cd federation-test
docker compose up -d synapse1

# Ждём ~30 секунд, пока сервер сгенерирует конфиг и стартанёт
curl http://localhost:8008/_matrix/client/versions
# Ожидаем JSON с версиями Matrix API — значит, сервер готов
```

Сервер слушает на `http://<ip-машины>:8008`. Чтобы мобильные устройства смогли подключиться:

- Узнать IP ноутбука в локальной сети: `ipconfig getifaddr en0` (mac) или `ip addr show` (linux).
- Телефон и ноутбук должны быть в **одной Wi‑Fi сети**.
- Если используете iPhone и HTTPS не настроен, Safari всё равно пустит (домен `localhost` и IP считаются insecure origin’ами, но WebCrypto для E2EE требует secure context — **используйте `localhost` либо поднимайте HTTPS через mkcert / Caddy**).

Для полноценного теста E2EE / Service Worker / push на мобильных **обязательно HTTPS**. Быстрый вариант:

```bash
# Пробросить localhost наружу через ngrok или cloudflared
npx cloudflared tunnel --url http://localhost:5173
# Или
ngrok http 5173
```

Получите URL вида `https://random.trycloudflare.com`, используйте его на всех устройствах.

#### 0.2.2. Вариант C: два Synapse (cross‑org)

Нужен только для проверок вроде «пользователь из нашей организации общается с пользователем контрагента». Детальная инструкция в `federation-test/README.md`. Для основного ручного прогона **не требуется**.

### 0.3. Регистрация тестовых аккаунтов

#### 0.3.1. Сводная таблица учётных данных

Создайте эти аккаунты перед прогоном. Все пароли одинаковые, потому что это изолированный тестовый сервер — в prod так делать нельзя.

| Логин | Пароль | Matrix ID (при `server1.localhost`) | Роль | На каких устройствах логиниться |
|---|---|---|---|---|
| `testuser1` | `testpass123` | `@testuser1:server1.localhost` | основной, отправитель | **D1**, **D1′**, **iOS1** (одни и те же креды на трёх устройствах) |
| `testuser2` | `testpass123` | `@testuser2:server1.localhost` | собеседник, получатель | **D2**, **iOS2**, **AND2** |
| `testuser3` | `testpass123` | `@testuser3:server1.localhost` | третий участник | **D3** |
| `testuser_throwaway` | `throwpass123` | `@testuser_throwaway:server1.localhost` | одноразовый для SET‑016 (deactivate) | создаётся перед кейсом, удаляется внутри него |
| `admin` | `adminpass123` | `@admin:server1.localhost` | admin homeserver для kick/ban/power‑level | любой свободный браузер (`ADM`) |

**Важно:** `D1`, `D1′`, `iOS1` — это **три разных устройства одного и того же аккаунта** `testuser1`. Логинитесь везде одним и тем же `testuser1 / testpass123`. Matrix создаст три независимые device‑сессии со своими ключами — именно эти сессии участвуют в cross‑sign в разделе 2.

Аналогично `testuser2` логинится на `D2`, `iOS2`, `AND2` — одним паролем, три device‑сессии.

#### 0.3.2. Команды регистрации

Для `testuser1/2/3` — обычный публичный эндпоинт (registration включена в `docker-compose.yml`):

```bash
for user in testuser1 testuser2 testuser3; do
  curl -s -X POST http://localhost:8008/_matrix/client/v3/register \
    -H "Content-Type: application/json" \
    -d "{\"username\":\"$user\",\"password\":\"testpass123\",\"auth\":{\"type\":\"m.login.dummy\"}}"
  echo
done
```

Для `admin` — через `register_new_matrix_user` внутри контейнера (нужен `-a` для admin‑прав):

```bash
docker exec -it synapse1 register_new_matrix_user \
  -c /data/homeserver.yaml \
  http://localhost:8008 \
  -u admin -p adminpass123 -a
```

`testuser_throwaway` **не создавайте заранее** — он нужен только для одного кейса и должен быть свежим. Создайте в момент прогона **SET‑016** (Privacy → Deactivate account):

```bash
curl -s -X POST http://localhost:8008/_matrix/client/v3/register \
  -H "Content-Type: application/json" \
  -d '{"username":"testuser_throwaway","password":"throwpass123","auth":{"type":"m.login.dummy"}}'
```

#### 0.3.3. Проверка, что аккаунты живы

Быстрая проверка, что все основные аккаунты залогиниваемы:

```bash
for user in testuser1 testuser2 testuser3; do
  echo -n "$user: "
  curl -s -X POST http://localhost:8008/_matrix/client/v3/login \
    -H "Content-Type: application/json" \
    -d "{\"type\":\"m.login.password\",\"user\":\"$user\",\"password\":\"testpass123\"}" \
    | grep -q access_token && echo OK || echo FAIL
done
```

Ожидаем три `OK`. Если `FAIL` — проверьте, что Synapse поднят и регистрация прошла.

### 0.4. Конфигурация клиента

Скопируйте `.env.example` в `.env`:

```bash
cp .env.example .env
```

Заполните:

```bash
# URL homeserver, к которому подключается приложение по умолчанию
# (пользователь может изменить его в ServerPicker на экране логина)
VITE_MATRIX_HOMESERVER_URL=http://localhost:8008

# Push‑ключи — нужны только для тестов push‑уведомлений (раздел 6).
# Если push не тестируете, оставьте пустыми.
VITE_VAPID_PUBLIC_KEY=
VITE_PUSH_GATEWAY_URL=http://localhost:3001
```

Перезапустите `npm run dev` — приложение будет по умолчанию использовать этот homeserver.

### 0.5. Push gateway (только для раздела 6)

Если тестируете push, поднимите локальный gateway:

```bash
# 1. Сгенерировать VAPID‑ключи один раз
cd push-gateway
npm install
npm run generate-keys
# Выведет VAPID_PUBLIC_KEY и VAPID_PRIVATE_KEY

# 2. Создать push-gateway/.env
cat > .env <<EOF
VAPID_PUBLIC_KEY=<ключ из вывода>
VAPID_PRIVATE_KEY=<приватный ключ из вывода>
VAPID_SUBJECT=mailto:admin@corp-matrix.local
PORT=3001
EOF

# 3. Запустить gateway
npm run dev
# Слушает http://localhost:3001
```

Скопируйте **public** ключ в корневой `.env` приложения:

```bash
VITE_VAPID_PUBLIC_KEY=<тот же public>
VITE_PUSH_GATEWAY_URL=http://localhost:3001
```

Чтобы Synapse доставлял уведомления в gateway, добавьте в `federation-test/synapse1-data/homeserver.yaml`:

```yaml
# В конец файла
app_service_config_files: []
push:
  include_content: true
```

Дальше пользователь получает push, когда приложение регистрирует подписку через `useNotifications` (UI: Настройки → Уведомления → «Включить»).

### 0.6. Распределение устройств и аккаунтов

Типичная раскладка для одного тестировщика. В сценариях ниже используются **короткие алиасы**, раскрытые в этой таблице:

| Алиас | Аккаунт | Устройство / браузер / окружение | Для чего |
|---|---|---|---|
| **D1** | `testuser1` | ноутбук, Chrome (обычное окно) | Большинство desktop‑кейсов, инициатор звонков/приглашений |
| **D2** | `testuser2` | ноутбук, Firefox (или 2‑й профиль Chrome) | Принимающая сторона, multi‑tab sync, второй участник |
| **D3** | `testuser3` | Chrome incognito | Group calls, 3‑й участник, свидетель истории при invite |
| **D1′** | `testuser1` | доп. browser profile / incognito на том же ноутбуке | Временная 3‑я сессия `testuser1` для multi‑device E2EE |
| **iOS1** | `testuser1` | iPhone, Safari + установленная PWA | iOS‑специфика, cross‑device sync для `testuser1` |
| **iOS2** | `testuser2` | второй iPhone (если есть) или Safari без PWA | Звонки/сообщения со стороны собеседника на iPhone |
| **AND2** | `testuser2` | Android, Chrome + установленная PWA | Android‑специфика, push, group calls со стороны `testuser2` |
| **ADM** | `admin` | любой desktop браузер | Kick/ban/power‑level кейсы (раздел 9.?) |

**Обозначения действий в сценариях:**

- `[D1 → D2]` — **D1** выполняет действие, **D2** получает/проверяет результат.
- `[D1 ↔ D2]` — двустороннее взаимодействие (оба действуют и наблюдают, напр. SAS‑верификация).
- `[D1 + iOS1]` — один пользователь на нескольких своих устройствах; наблюдение синхронно.
- `[iOS1]` — действие и проверка на одном устройстве одним пользователем.
- `[iOS1 | AND2]` — допустимо любое из двух; полный прогон желательно на обоих.

**Важно для E2EE:** каждое устройство создаёт **собственную** Matrix‑сессию (device). Для `testuser1` получится до 3 сессий (D1 + iOS1 + D1′). Именно их нужно cross‑sign в разделе 2.

**Важно для звонков:** `testuser1` и `testuser2` должны быть в одной DM‑комнате. Создайте её один раз:

1. На **D1** залогиниться как `testuser1`.
2. Создать DM с `@testuser2:server1.localhost`.
3. На **D2** принять приглашение.
4. Эту DM использовать во всех кейсах `D‑CALL‑*` и `M‑CALL‑*`.

Для group calls дополнительно создайте комнату **«Test Group Room»** с участниками `testuser1`, `testuser2`, `testuser3`.

### 0.7. Сбор данных между прогонами

Перед каждым полным прогоном желательно сбросить состояние, чтобы не накапливались артефакты:

```bash
# Остановить и удалить homeserver вместе с данными
cd federation-test
docker compose down -v
rm -rf synapse1-data

# Поднять заново — автоматически регенерирует конфиг
docker compose up -d synapse1

# Зарегистрировать пользователей заново (скрипт из 0.3)
```

Если сбрасывать не хотите — после каждого прогона хотя бы удаляйте одноразовые комнаты и `testuser_throwaway`.

На самих устройствах между прогонами:
- Desktop: DevTools → Application → Clear site data.
- iPhone: Настройки → Safari → Очистить историю и данные (для всей Safari) **или** удалить и переустановить PWA.
- Android Chrome: Настройки → Конфиденциальность → Очистить данные просмотра (для конкретного сайта в Site settings).

## 1. WebRTC‑звонки

Автотест проверит только, что UI появился. Качество медиапотока, задержка, эхо, переключение устройств — только руками.

### 1.1. Desktop → Desktop (один и тот же homeserver)

- [ ] **D‑CALL‑01** `[D1 → D2]` Исходящий голосовой в DM: `D1` инициирует из RoomHeader. Звук чистый в обе стороны, задержка < 300 мс, нет эха.
- [ ] **D‑CALL‑02** `[D1 → D2]` Исходящий видео: `D1` кликает «Видео», `D2` принимает. Обе камеры активны, картинка не фризит ≥ 30 с.
- [ ] **D‑CALL‑03** `[D1 → D2]` Mute mic на `D1` → `D2` слышит тишину, не помехи.
- [ ] **D‑CALL‑04** `[D1 → D2]` Camera off на `D1` → у `D2` чёрный экран или placeholder, но звук сохраняется.
- [ ] **D‑CALL‑05** `[D1]` Во время звонка подключить Bluetooth‑гарнитуру к ноутбуку `D1` → звук переключается на неё без разрывов.
- [ ] **D‑CALL‑06** `[D1]` Отключить и заново подключить BT‑гарнитуру → вход/выход звука не зависает.
- [ ] **D‑CALL‑07** `[D1]` В браузере отозвать permission на микрофон (Settings → Site settings → reset) → повторный call инициируется, UI graceful, показан toast, call завершается корректно.
- [ ] **D‑CALL‑08** `[D1 → D2]` End call одной стороной → у обоих CallView закрывается за < 2 с.
- [ ] **D‑CALL‑09** `[D1 + D2 + D3]` Group call в «Test Group Room»: все трое подключаются, все тайлы живые, нет фризов при выходе `D3`.
- [ ] **D‑CALL‑10** `[D1]` Вторая камера (USB webcam): в настройках системы сделать её default → переключиться на неё из браузера, увидеть корректное видео.

### 1.2. Mobile звонки

- [ ] **M‑CALL‑01** `[iOS1 → AND2]` Исходящий голосовой из DM с iPhone → Android. Чистый звук в обе стороны.
- [ ] **M‑CALL‑02** `[iOS1 → AND2]` Видео‑звонок. На обоих фронтальная камера активируется сразу, ориентация картинки правильная.
- [ ] **M‑CALL‑03** `[iOS1 | AND2]` Переключение между фронтальной и задней камерой (если кнопка есть) — без разрыва.
- [ ] **M‑CALL‑04** `[iOS1 | AND2]` Поворот устройства во время видео‑звонка → картинка не переворачивается вверх ногами.
- [ ] **M‑CALL‑05** `[iOS1]` Во время WebRTC‑звонка с `AND2` на `iOS1` приходит реальный телефонный звонок на SIM → приложение либо пауза, либо graceful hangup, никакого «залипания».
- [ ] **M‑CALL‑06** `[iOS1 | AND2]` Перевод приложения в фон во время звонка (home button) → звук сохраняется или корректно прерывается; возврат в foreground не ломает состояние.
- [ ] **M‑CALL‑07** `[iOS1 | AND2]` Блокировка экрана во время звонка (power button) → звук продолжается, разблокировка возвращает UI.
- [ ] **M‑CALL‑08** `[iOS1]` AirPods / Bluetooth‑гарнитура на iPhone → звук маршрутизируется на гарнитуру, mic гарнитуры работает.
- [ ] **M‑CALL‑09** `[D1 → AND2]` PWA на `AND2` свёрнута или закрыта, `D1` звонит → доставляется push‑уведомление с accept/reject (если реализовано через push).
- [ ] **M‑CALL‑10** `[iOS1 → AND2]` `iOS1` в зоне слабого 4G (1 bar) → UI показывает деградацию соединения; не молчание без индикации.

## 2. E2EE — ceremonies с участием человека

Флоу автоматизируется, но ключевые сравнения (SAS‑эмодзи, фразы) — глазами.

### 2.1. SAS cross‑sign verification

- [ ] **E2EE‑01** `[D1 ↔ D2]` В комнате между `testuser1` и `testuser2` запустить верификацию через UserProfilePopup → `CrossSignVerification`. Оба видят **одинаковые** 7 эмодзи в том же порядке. Подтвердить.
- [ ] **E2EE‑02** `[D1 ↔ D2]` После подтверждения — в RoomHeader у обоих горит EncryptionBadge «verified» (зелёный/галочка).
- [ ] **E2EE‑03** `[D1 ↔ D2]` Повторная верификация. `D1` выбирает «Не совпадают» (имитация MITM) → верификация прерывается, badge остаётся неверифицированным.
- [ ] **E2EE‑04** `[D1 → D2]` `D1` инициирует верификацию → `IncomingVerificationDialog` всплывает на `D2` за < 3 с.
- [ ] **E2EE‑05** `[D1 ↔ iOS1]` Self‑verify: `testuser1` верифицирует свою iOS‑сессию с desktop‑сессии. SAS совпадает, оба устройства отмечены как verified в DevicesSettings.

### 2.2. Key backup

- [ ] **E2EE‑06** `[D1]` Настройки → Encryption → KeyBackupSetup: создать бэкап с passphrase. Записать passphrase.
- [ ] **E2EE‑07** `[D1 → D2]` `D1` отправляет 10 зашифрованных сообщений в комнату с `D2`, подождать > 1 минуты (чтобы ключи ушли в бэкап).
- [ ] **E2EE‑08** `[D1]` Logout с `D1` → login заново как `testuser1` → сообщения из шага `E2EE‑07` **не расшифрованы** (ожидаемо, пока ключи не восстановлены).
- [ ] **E2EE‑09** `[D1]` `KeyRestoreScreen` → ввести passphrase из `E2EE‑06` → исторические сообщения расшифровались корректно.
- [ ] **E2EE‑10** `[D1]` Logout → login → ввести **неверную** passphrase 3 раза → graceful error, не краш, не race.
- [ ] **E2EE‑11** `[D1]` Настройки → Encryption → экспорт ключей в .txt. Logout → login → `ImportKeysDialog` → импорт .txt → исторические сообщения расшифровываются.
- [ ] **E2EE‑12** `[iOS1 → D1]` С `iOS1` отправить сообщение в encrypted room. На `D1` (если ключ ещё не пришёл) — появляется placeholder «Unable to decrypt». Через sync + backup — расшифровывается, либо остаётся UTD по ожидаемой причине, но без краша.

### 2.3. Многоустроистные сценарии

- [ ] **E2EE‑13** `[D1 + iOS1 + D1′]` 3 сессии `testuser1`: сделать cross‑sign по цепочке (D1 ↔ iOS1, потом D1 ↔ D1′). На всех трёх EncryptionBadge «verified», все три видны как verified в DevicesSettings любого из них.
- [ ] **E2EE‑14** `[D1 → D3]` В E2EE комнате с `history_visibility=shared` (выставить через AccessibilitySettings на `D1`) пригласить `testuser3`. На `D3` принять приглашение → `testuser3` видит предыдущую историю (ключи получены).
- [ ] **E2EE‑15** `[D1 → D3]` То же, но `history_visibility=joined`. На `D3` — **только** UTD‑placeholder для старых сообщений; новые приходят нормально.

## 3. PWA на реальных устройствах

Полный install flow автоматизировать нельзя — ни в iOS, ни с реальным A2HS баннером Android.

### 3.1. iOS Safari

Весь раздел: аккаунт `testuser1`, устройство **iOS1**.
